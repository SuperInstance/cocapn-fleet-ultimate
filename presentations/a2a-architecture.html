Protocol

What is A2A?

A2A (Agent-to-Agent) is Google's open protocol for autonomous agents to discover, negotiate, and collaborate.

🎯 Design Goal

Any agent from any vendor should be able to work with any other agent, regardless of underlying model or framework.

📜 Open Standard

Published by Google, Apache 2.0 licensed. Not a product — a protocol anyone can implement.

A2A is an open protocol, not a product. Six stages: discovery, authentication, negotiation, delegation, execution, reporting. The hexagon layout in the SVG shows this cycle.

Discovery

Agent Cards — Self-Describing Agents

Every fleet agent exposes an Agent Card at a well-known URL. Think of it as a resume + API spec + personality profile.

🔍 Discovery Flow

1. Agent A queries directory or scans known hosts
2. Fetches /.well-known/agent.json
3. Parses capabilities, skills, auth requirements
4. Decides if delegation makes sense

Delegation

Task Delegation Flow

1. SEND

Agent A POSTs task to Agent B

→
2. ACK

B validates, creates task ID

→
3. STREAM

Real-time status + artifacts

→
4. RESOLVE

Final output or error

The delegation flow is four steps with a central Task object. Agent A sends a task, B acknowledges with a task ID, then streams real-time updates via Server-Sent Events. The Task object carries status, artifacts, and full history. Human-in-the-loop is built-in, not bolted on.

Integration

FLUX as an A2A Agent

FLUX isn't just a VM — it's an A2A-first agent that advertises its capabilities via an Agent Card.

🎴 Agent Card Skills

  • verify — Formal proof checking
  • execute — Deterministic bytecode
  • schedule — Real-time task scheduling
  • monitor — WCET analysis

⚡ Why This Matters

Other agents can delegate safety-critical computation to FLUX without knowing how proofs work. They just send an A2A task.

FLUX exposes itself as an A2A agent. Other agents send proof obligations as task artifacts, FLUX verifies them using its formally-proven engine, and returns results as structured artifacts. The calling agent doesn't need to understand Coq or proof theory.
Edge

Hardware via A2A — The Edge Mesh

Every device runs an A2A agent. The Edge Gateway routes between local and cloud. Sensors and actuators speak A2A via Zerolang.

The hardware mesh: every device is an A2A agent. Jetson runs FLUX for verified computation. Pi runs full agents. MCU-class devices run Zerolang. The Edge Gateway bridges local mesh to cloud. All communication is A2A — no custom protocols needed.
Security

Security Considerations

🔐

Authentication

OAuth 2.0 + mTLS for all A2A traffic. Agent Cards declare supported schemes. Tokens rotate every 15 minutes.

🛡️

Authorization

Capability-based access control. Agents get fine-grained permissions per skill, not blanket access.

🔏

Encryption

E2EE for sensitive artifacts. Task metadata is signed. FLUX proofs are cryptographically verifiable.

⚠️

Sandboxing

Each agent runs in an isolated container. FLUX provides formally-verified memory isolation. No shared state.

Security is not an afterthought — it's a first-class design constraint in the A2A spec.

Four security layers. Auth: OAuth + mTLS. Authz: capability-based, not role-based — agents get per-skill permissions. Encryption: end-to-end for artifacts, signed metadata. Sandboxing: formal isolation from FLUX, container isolation for agents.
